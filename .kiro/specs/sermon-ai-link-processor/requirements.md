# Requirements Document — Sermon AI Link Processor

## Introduction

The Sermon AI Link Processor is a feature addition to the KDC Uganda CMS Platform's Sermons CRUD Manager. It enables administrators and editors to paste YouTube and social media links, automatically extract and transcribe video content using free AI services, generate sermon summaries, apply SEO optimization, and review the generated content in a draft form before publishing.

This feature integrates seamlessly with the existing sermon workflow, leveraging the sermon fields (title, description, content, video_url, etc.) and maintaining the draft-review-publish pattern already established in the CMS.

---

## Glossary

- **Sermon_AI_Processor**: The system component that accepts media links, orchestrates AI processing, and returns generated content.
- **Media_Link**: A URL pointing to a YouTube video or social media post containing sermon content (e.g., `https://www.youtube.com/watch?v=...`).
- **Transcription**: The text output generated by converting audio/video content to written form.
- **Sermon_Summary**: A concise, AI-generated overview of the sermon's key points and themes.
- **SEO_Optimization**: The process of generating or enhancing title, description, and keywords to improve search engine visibility.
- **Draft_Form**: A pre-populated sermon form displaying AI-generated content for user review and editing before saving.
- **Free_AI_Service**: An AI API or service that provides transcription, summarization, or text generation without requiring paid API calls (e.g., Hugging Face, Ollama, or similar open-source models).
- **Link_Validation**: The process of verifying that a pasted URL is a valid, supported media link.
- **Rate_Limiting**: A mechanism to prevent excessive API calls and ensure fair resource usage.
- **Processing_Status**: The current state of a link being processed (pending, processing, completed, failed).
- **Error_Recovery**: The ability to handle and report processing failures gracefully without corrupting the sermon record.
- **Sermons_CRUD_Manager**: The existing admin interface at `/admin/sermons` for creating, reading, updating, and deleting sermon records.
- **Server_Action**: A Next.js 14 server-side async function used for mutations and background processing.
- **Supabase_Client**: The browser-side or server-side Supabase client for database operations.
- **Admin_User**: A user with role `admin` or `editor` who can access the Sermons CRUD Manager.

## Requirements

### Requirement 1: Link Input Interface

**User Story:** As an editor or admin, I want to paste a YouTube or social media link into the sermon form, so that I can quickly populate sermon content from existing video sources.

#### Acceptance Criteria

1. THE Sermons_Manager form SHALL display a new "AI Link Processor" section above the existing sermon fields.
2. THE AI Link Processor section SHALL contain a text input field labeled "Paste Media Link" with placeholder text "https://www.youtube.com/watch?v=...".
3. WHEN an Admin_User pastes a URL into the link input field, THE Sermons_Manager SHALL validate the URL format client-side and display an inline error if the URL does not match a supported pattern (YouTube, Vimeo, or direct video file).
4. WHEN an Admin_User clicks a "Process Link" button, THE Sermons_Manager SHALL disable the button, display a loading indicator, and invoke a Server_Action to begin processing.
5. IF the link input is empty or contains only whitespace, THEN THE "Process Link" button SHALL remain disabled and display a tooltip "Please enter a valid link".
6. THE link input field SHALL be cleared after successful processing or when the user manually clears it.

---

### Requirement 2: Link Validation

**User Story:** As the system, I want to validate media links before processing, so that invalid or unsupported URLs are rejected early with clear feedback.

#### Acceptance Criteria

1. THE Sermon_AI_Processor SHALL validate that the pasted URL matches one of the following patterns:
   - YouTube: `youtube.com/watch?v=*` or `youtu.be/*`
   - Vimeo: `vimeo.com/*`
   - Direct video file: URLs ending in `.mp4`, `.webm`, `.ogg`, or `.mov`
2. WHEN a URL does not match any supported pattern, THE Server_Action SHALL return an error response with a user-friendly message: "Unsupported link format. Please paste a YouTube, Vimeo, or direct video file URL."
3. WHEN a URL matches a supported pattern but the video is inaccessible (e.g., deleted, private, or region-blocked), THE Server_Action SHALL attempt to process it and return a descriptive error if the video cannot be accessed.
4. THE link validation SHALL be performed server-side before any AI processing begins.

---

### Requirement 3: Free AI Transcription

**User Story:** As the system, I want to transcribe video content using free AI services, so that sermon audio is converted to text without incurring paid API costs.

#### Acceptance Criteria

1. WHEN a valid media link is submitted, THE Sermon_AI_Processor SHALL extract audio from the video using a free tool (e.g., `yt-dlp` for YouTube, or equivalent).
2. THE Sermon_AI_Processor SHALL transcribe the extracted audio using a free AI model (e.g., OpenAI Whisper via Hugging Face Inference API, or a self-hosted Ollama instance).
3. IF transcription succeeds, THE Server_Action SHALL store the transcription text in memory and proceed to summarization.
4. IF transcription fails (e.g., audio extraction error, API timeout, unsupported audio format), THE Server_Action SHALL return an error response with a message: "Failed to transcribe video. Please try again or manually enter the sermon content."
5. THE transcription process SHALL have a timeout of 5 minutes per video to prevent indefinite processing.
6. THE Sermon_AI_Processor SHALL NOT store raw transcriptions in the database; transcriptions are used only to generate summaries and SEO content.

---

### Requirement 4: Free AI Summarization

**User Story:** As the system, I want to generate comprehensive sermon summaries from transcribed content, so that detailed sermon content is automatically extracted without manual effort.

#### Acceptance Criteria

1. WHEN transcription is complete, THE Sermon_AI_Processor SHALL pass the transcription text to a free AI summarization model (e.g., Google Gemini, Hugging Face Transformers, Ollama, or similar).
2. THE Sermon_AI_Processor SHALL generate a comprehensive summary of minimum 1500 words capturing the main themes, key scripture references, actionable takeaways, sermon structure, and detailed explanations of key points.
3. IF summarization succeeds, THE Server_Action SHALL store the summary text in memory for display in the Draft_Form.
4. IF summarization fails, THE Server_Action SHALL return an error response with a message: "Failed to generate summary. Please try again or manually enter the sermon summary."
5. THE summarization process SHALL have a timeout of 5 minutes to prevent indefinite processing (increased from 2 minutes to accommodate longer content generation).
6. THE generated summary SHALL be presented to the user for review and editing before saving.

---

### Requirement 5: Free AI SEO Optimization

**User Story:** As the system, I want to generate SEO-optimized titles, descriptions, and keywords, so that sermons are discoverable via search engines without manual SEO work.

#### Acceptance Criteria

1. WHEN summarization is complete, THE Sermon_AI_Processor SHALL generate an SEO-optimized title (50–70 characters) based on the sermon content and transcription.
2. THE Sermon_AI_Processor SHALL generate an SEO-optimized description (150–160 characters) suitable for search engine snippets.
3. THE Sermon_AI_Processor SHALL generate 5–8 relevant keywords or key phrases (comma-separated) related to the sermon topic.
4. IF SEO generation succeeds, THE Server_Action SHALL store the title, description, and keywords in memory for display in the Draft_Form.
5. IF SEO generation fails, THE Server_Action SHALL return an error response with a message: "Failed to generate SEO content. Please try again or manually enter SEO details."
6. THE SEO generation process SHALL have a timeout of 1 minute.
7. THE generated SEO content SHALL be presented to the user for review and editing before saving.

---

### Requirement 6: Draft Form Display

**User Story:** As an editor or admin, I want to review and edit AI-generated content in a draft form, so that I can ensure accuracy and quality before publishing.

#### Acceptance Criteria

1. WHEN link processing completes successfully, THE Sermons_Manager SHALL display a "Draft Review" modal or panel containing pre-populated fields:
   - Title (editable, from SEO generation)
   - Description (editable, from SEO generation)
   - Content (editable, from summarization)
   - Keywords (editable, from SEO generation)
   - Video URL (pre-filled with the original link)
   - Preacher (editable, empty or pre-filled if available)
   - Series (editable, empty or pre-filled if available)
   - Date (editable, defaults to today's date)
   - Duration (editable, optional)
   - Thumbnail (Media_Library picker, optional)
   - Status (defaults to `draft`)
2. THE Draft_Form SHALL display a "Full Transcription" section (collapsible) showing the complete transcription text for reference.
3. THE Draft_Form SHALL provide "Edit", "Save as Draft", and "Publish" buttons.
4. WHEN the user clicks "Edit", THE Draft_Form SHALL allow inline editing of all fields.
5. WHEN the user clicks "Save as Draft", THE Server_Action SHALL save the sermon record with `status = 'draft'` and display a success message.
6. WHEN the user clicks "Publish", THE Server_Action SHALL save the sermon record with `status = 'published'` and set `published_at` to the current UTC timestamp.
7. WHEN the user closes the Draft_Form without saving, THE Sermons_Manager SHALL discard the generated content and return to the link input state.

---

### Requirement 7: Integration with Existing Sermon Fields

**User Story:** As the system, I want the AI processor to work seamlessly with existing sermon fields, so that the feature integrates naturally into the current workflow.

#### Acceptance Criteria

1. THE Sermon_AI_Processor SHALL populate the following existing sermon fields from AI-generated content:
   - `title` (from SEO title)
   - `description` (from SEO description)
   - `content` (from summarization)
   - `video_url` (from the pasted link)
   - `status` (defaults to `draft`)
2. THE Sermon_AI_Processor SHALL NOT override the following fields unless explicitly provided by the user:
   - `preacher` (user must enter or select)
   - `series` (user must enter or select)
   - `date` (defaults to today, user can edit)
   - `duration_minutes` (optional, user can enter)
   - `thumbnail_url` (user can select from Media_Library)
3. WHEN a sermon is saved from the Draft_Form, THE Server_Action SHALL call `revalidatePath('/sermons')` and `revalidatePath('/sermons/[slug]')` to invalidate the public cache.
4. THE slug field SHALL be auto-generated from the title using the existing `generateSlug` utility and SHALL be editable by the user.

---

### Requirement 8: Rate Limiting and Performance

**User Story:** As the system, I want to prevent abuse and ensure fair resource usage, so that the AI processor remains responsive for all users.

#### Acceptance Criteria

1. THE Sermon_AI_Processor SHALL implement rate limiting to allow a maximum of 5 link processing requests per Admin_User per hour.
2. WHEN an Admin_User exceeds the rate limit, THE Server_Action SHALL return an error response with a message: "You have reached the maximum number of processing requests this hour. Please try again later."
3. THE rate limit counter SHALL be stored in the database (e.g., in a `processing_logs` table) and reset hourly.
4. THE Sermon_AI_Processor SHALL process links asynchronously (not blocking the user's browser) using a Server_Action or background job.
5. THE processing status (pending, processing, completed, failed) SHALL be displayed to the user in real-time via a status indicator or progress bar.
6. THE Sermon_AI_Processor SHALL have a total processing timeout of 10 minutes per link; if processing exceeds this time, the request SHALL be cancelled and an error returned.

---

### Requirement 9: Error Handling and Recovery

**User Story:** As an editor or admin, I want clear error messages and recovery options, so that I can troubleshoot processing failures and retry without losing data.

#### Acceptance Criteria

1. WHEN any step of the processing pipeline fails (link validation, transcription, summarization, SEO generation), THE Server_Action SHALL return a descriptive error message indicating which step failed.
2. THE error message SHALL include a suggestion for recovery (e.g., "Try a different video", "Check your internet connection", "Try again in a few minutes").
3. WHEN processing fails, THE Sermons_Manager SHALL display the error in a dismissible alert and allow the user to:
   - Retry the same link
   - Clear the link and try a different one
   - Manually enter sermon content instead
4. THE Sermon_AI_Processor SHALL NOT create a partial or corrupted sermon record if processing fails; the database state SHALL remain unchanged.
5. WHEN a processing request times out, THE Server_Action SHALL return an error response with a message: "Processing took too long. Please try again or manually enter the sermon content."

---

### Requirement 10: Free AI Service Integration

**User Story:** As a developer, I want to use free AI services, so that the feature operates without paid API costs.

#### Acceptance Criteria

1. THE Sermon_AI_Processor SHALL use one of the following free AI services (or equivalent):
   - **Transcription**: OpenAI Whisper via Hugging Face Inference API (free tier), or self-hosted Ollama with a speech-to-text model
   - **Summarization**: Hugging Face Transformers (facebook/bart-large-cnn or similar), or self-hosted Ollama with a summarization model
   - **SEO Generation**: Hugging Face Transformers (text generation model), or self-hosted Ollama with a text generation model
2. THE Sermon_AI_Processor SHALL NOT use paid APIs (e.g., OpenAI GPT-4, Google Cloud Speech-to-Text, AWS Transcribe) without explicit user configuration and consent.
3. THE Sermon_AI_Processor configuration (API endpoints, model names, API keys) SHALL be stored in environment variables and documented in a `.env.example` file.
4. IF a free AI service is unavailable or returns an error, THE Server_Action SHALL return a user-friendly error message and SHALL NOT fall back to a paid service without explicit user approval.

---

### Requirement 11: Logging and Monitoring

**User Story:** As a developer, I want to log processing events and monitor performance, so that I can debug issues and optimize the system.

#### Acceptance Criteria

1. THE Sermon_AI_Processor SHALL log the following events server-side:
   - Link submission (timestamp, user ID, link URL)
   - Processing start (timestamp, processing step)
   - Processing completion (timestamp, duration, success/failure)
   - Processing errors (timestamp, error message, step where error occurred)
2. THE logs SHALL be stored in a `processing_logs` table with columns: `id`, `user_id`, `link_url`, `status`, `error_message`, `duration_ms`, `created_at`.
3. THE logs SHALL be retained for at least 30 days for debugging and audit purposes.
4. THE Sermon_AI_Processor SHALL measure and log the duration of each processing step (transcription, summarization, SEO generation) for performance monitoring.
5. IF processing duration exceeds expected thresholds (e.g., transcription > 3 minutes), THE system SHALL log a warning for investigation.

---

### Requirement 12: User Feedback and Status Indication

**User Story:** As an editor or admin, I want real-time feedback on processing status, so that I know what is happening and when to expect results.

#### Acceptance Criteria

1. WHEN link processing begins, THE Sermons_Manager SHALL display a loading indicator with the message "Processing video... (Step 1 of 3: Transcribing)".
2. AS each processing step completes, THE loading indicator SHALL update to show the current step (e.g., "Step 2 of 3: Summarizing", "Step 3 of 3: Optimizing for SEO").
3. WHEN processing completes successfully, THE loading indicator SHALL be replaced with the Draft_Form containing the generated content.
4. WHEN processing fails, THE loading indicator SHALL be replaced with an error message and recovery options.
5. THE Sermons_Manager SHALL display an estimated time remaining (e.g., "Estimated time: 2 minutes") based on historical processing durations.
6. THE user SHALL be able to cancel a processing request at any time by clicking a "Cancel" button; cancellation SHALL stop the processing and return to the link input state.

---

### Requirement 13: Accessibility and Usability

**User Story:** As an editor with accessibility needs, I want the AI processor to be accessible and easy to use, so that I can process sermons efficiently.

#### Acceptance Criteria

1. THE AI Link Processor section SHALL have a clear, descriptive heading (e.g., "AI-Powered Sermon Generator").
2. ALL form inputs (link input, edit fields in Draft_Form) SHALL have associated `<label>` elements with descriptive text.
3. THE loading indicator and status messages SHALL be announced to screen readers using ARIA live regions.
4. THE Draft_Form modal SHALL be keyboard-navigable with proper focus management and tab order.
5. ALL buttons (Process Link, Save as Draft, Publish, Cancel) SHALL have descriptive labels and be keyboard-accessible.
6. THE error messages SHALL be displayed in a color-independent way (not relying solely on red text) and include an icon or text prefix (e.g., "Error:").

---

### Requirement 14: Security and Data Privacy

**User Story:** As an admin, I want the AI processor to handle data securely, so that sermon content and user data are protected.

#### Acceptance Criteria

1. THE Sermon_AI_Processor SHALL validate and sanitize all user inputs (link URL, edited fields) to prevent injection attacks.
2. THE Sermon_AI_Processor SHALL NOT store raw transcriptions in the database; transcriptions SHALL be discarded after summarization and SEO generation are complete.
3. THE Sermon_AI_Processor SHALL NOT send sermon content or user data to external AI services beyond what is necessary for processing (e.g., transcription audio, summarization text).
4. THE Sermon_AI_Processor SHALL use HTTPS for all external API calls to free AI services.
5. THE Server_Action SHALL verify the user's role (`admin` or `editor`) before allowing link processing; users with role `author` or `member` SHALL NOT be able to use the AI processor.
6. THE processing logs SHALL NOT contain sensitive data (e.g., full sermon content, user passwords); logs SHALL only contain metadata (user ID, link URL, status, error message).

---

### Requirement 15: Configuration and Deployment

**User Story:** As a developer, I want to configure the AI processor for different deployment environments, so that the feature works in development, staging, and production.

#### Acceptance Criteria

1. THE Sermon_AI_Processor configuration SHALL be stored in environment variables:
   - `AI_TRANSCRIPTION_SERVICE` (e.g., `huggingface`, `ollama`)
   - `AI_TRANSCRIPTION_API_KEY` (if using Hugging Face)
   - `AI_TRANSCRIPTION_ENDPOINT` (if using self-hosted Ollama)
   - `AI_SUMMARIZATION_SERVICE` (e.g., `huggingface`, `ollama`)
   - `AI_SUMMARIZATION_ENDPOINT` (if using self-hosted Ollama)
   - `AI_SEO_SERVICE` (e.g., `huggingface`, `ollama`)
   - `AI_SEO_ENDPOINT` (if using self-hosted Ollama)
   - `AI_PROCESSING_TIMEOUT_MS` (default: 600000 = 10 minutes)
   - `AI_RATE_LIMIT_PER_HOUR` (default: 5)
2. THE `.env.example` file SHALL document all configuration variables with example values.
3. THE Sermon_AI_Processor SHALL gracefully degrade if a free AI service is unavailable; the user SHALL be informed and offered manual entry as an alternative.
4. THE Sermon_AI_Processor SHALL support both cloud-based (Hugging Face) and self-hosted (Ollama) AI services without code changes, only configuration changes.
